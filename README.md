# 石听 NeoLoop

> 让每条反馈都找到一辆车、一个任务、一个责任人和一次改进。

石听 NeoLoop 是“石听未来”为新石器企业命题准备的反馈闭环协同方案。它不接管车辆运行，也不重复查车、发车、开门、充电调度等既有运营能力；它做的是把分散的用户反馈、客服记录和现场信息，组织为可追踪、可复盘的改进闭环。

备用静态演示：[https://shiting-neoloop-demo.netlify.app](https://shiting-neoloop-demo.netlify.app)。仓库中的 React 版本提供完整的点击演示；全部业务内容仍为合成样例，不依赖企业生产系统。

## 这个仓库展示什么

- 一个可点击的反馈闭环演示：产品概览、合成案件、问题簇和带人工关口的处置路径；
- 一个独立的参赛团队页面，只展示本次公开范围内的姓名、学校、专业和团队角色；参赛成员不映射为企业流程执行者；
- 一份公开可读的方案边界：哪些是公开事实，哪些仍待企业验证；
- 一组仅供演示的合成案例，不含真实客户、车辆、事故、运营数据或经营结论；
- 可直接作为 GitHub Pages 来源目录的 `docs/` 静态站点。

## 方案边界

| 内容类型 | 在本仓库中的处理 |
| --- | --- |
| 公开事实 | 仅用于说明项目背景，不据此推断新石器内部流程或数据状态。 |
| 待企业验证 | 以“待验证”表达，例如字段可得性、派单规则、责任边界。 |
| 方案设计 | 呈现为候选流程和界面原型，等待企业侧确认后落地。 |
| 合成实验 | 所有案例、车辆标识、任务编号、时长、数量和指标均为虚构样例。 |

NeoLoop 不执行车辆控制；不自动判定事故责任、赔偿、恢复时间或最终技术根因。关键分级、责任人确认和结案均保留人工审核。

## 本地预览

```bash
npm install
npm run dev
```

打开终端给出的本地地址即可查看 React 交互版。GitHub Pages 使用 `docs/` 下的零构建静态版本；直接打开 `docs/index.html` 或运行任意静态文件服务器也可以查看。

## 发布到 GitHub Pages（可选）

1. 进入仓库 **Settings → Pages**，选择 **Deploy from a branch**。
2. 选择 `main` 分支与 `/docs` 文件夹，保存后等待 GitHub 生成地址。
3. 将生成的网址放到个人主页 README 的置顶项目中。

完整操作见 [docs/GITHUB_PAGES_SETUP.md](docs/GITHUB_PAGES_SETUP.md)。提交或公开前，请由参赛者确认赛事条款、个人信息和可公开范围。

## 仓库结构

```text
app/                  React 交互版源文件
docs/                 GitHub Pages 静态站点与公开说明
data/demo-cases.json  合成案例数据

```
